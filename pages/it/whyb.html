# 前言

有数据库基础的应该都了解到数据库利用索引能达到一个快速的找寻数据的效果。

更深一点的可能了解到索引(InnoDB)是利用B+树去实现的。

但是为什么要用B+树呢？

这世上树那么多,为何独爱这一棵呢？

一切都源于硬件。

# CPU如何处理数据

CPU要处理数据必须从内存中获取,如果内存中没有,那么就要去硬盘上查找了(一般这个查找会交给DMA去处理,CPU可以去做其他事情)。

这个从硬盘查找数据的过程就是IO了。

但是IO是有成本的,不管是什么硬盘都会存在延时。

为什么会有延时呢？ 因为数据放在硬盘里,你需要去找到它并且把它带回内存。

当然CPU和内存的交互不是这么简单的，这里只是说一个大体流程。

所以, 发生IO的次数越少我们应用的性能就越高。

那么, InnoDB引擎在设计时就需要尽可能的保证IO次数少并且支持的数据量还要尽可能的大。

# 索引数据结构的选择

# Hash

Hash结构确实搜索可以达到O(1)的效果, 但是光不支持范围搜索就毙掉了。

# 二叉树

  • 二叉搜索树

特点是左边小于根节点,右边大于根节点,但是插入新数据不会使之旋转。

这也就导致了一个问题, 插入的数据连续时会退化成一个链表,任何搜索都是扫全表.

  • AVL搜索树

也叫强平衡二叉搜索树，特点左右子树的高度相差不能超过1.

这也就导致了它会频繁的发生旋转,影响写入的性能。

  • 红黑树

特点红色节点的子节点必须是黑色。

每个节点到子节点都必须经过相同的黑色点

这样可以保证不会旋转太频繁,但是树高是不可控的。

树高可以看做是IO次数。

# 多叉树

  • B树

B树的特点是每个节点都会保存数据.

  • B+树

B+树的特点是只有叶子节点会保存数据。

  • B*树

InnoDB改造后的B+树,每个节点都保存了指向相邻节点的指针。

# 总结

任何不考虑硬件特性的程序都是耍流氓。
